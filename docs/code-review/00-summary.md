## 4. 用户反馈问题分析

### 4.1 "没到 7 天就过期了"

**可能原因**:
1. 在其他设备登出导致 Refresh Token 被吊销
2. 用户本地时间不准确
3. 使用隐私模式浏览器

**排查建议**:
- 检查 Redis 中是否存在对应的黑名单记录
- 检查用户本地时间与服务器时间差异

### 4.2 "到期后没有自动退出" ✅ 已修复

**根本原因**:
- 前端只在路由切换或 API 请求时检查 Token 有效性
- 用户停留在同一页面不会触发检查

**修复方案（已实施）**:
- 新增 `useTokenGuard` composable，每 5 分钟定时检查 Token 有效期
- 监听 `visibilitychange` 事件，页面切回前台时立即检查
- Token 即将过期时主动刷新，刷新失败则自动登出清除本地状态

## 5. 代码质量亮点

### ✅ 优秀实践

1. **完善的黑名单机制**
   - 登出时将 Token JTI 加入 Redis 黑名单
   - 防止已登出 Token 被滥用

2. **并发刷新保护**
   - 前端使用 `isRefreshing` 标志和订阅队列
   - 防止多个请求同时触发刷新

3. **灵活的规则系统**
   - 支持 mandatory/default_on/default_off/custom 四种作用域
   - mandatory 规则不可被用户覆盖

4. **脏数据自动清理**
   - 租户删除规则后，用户覆盖自动过滤
   - 避免产生孤立数据

5. **完善的权限控制**
   - 用户权限锁定机制（AllowCustomFields/Rules/Strictness）
   - 访问控制白名单机制

6. **主动 Token 过期检测**（新增）
   - TokenGuard 定时 + visibilitychange 双重机制
   - 确保 Token 过期被及时发现并处理

7. **配置统一读取**（新增）
   - Token TTL 优先从数据库 system_configs 读取
   - 管理员修改系统设置后立即生效，降级使用 config.yaml

8. **后端错误消息国际化**（新增）
   - 后端返回标准错误码 + 英文 fallback 消息
   - 前端 `ERROR_CODE_I18N_MAP` 按错误码映射 i18n 键
   - 支持 zh-CN / en-US 双语错误提示

9. **默认密码可配置化**（新增）
   - 从 `system_configs.auth.default_password` 读取
   - 管理员可在系统设置中随时修改，无需重启服务
